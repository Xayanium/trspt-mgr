# 校园机动车管理系统业务功能测试整理

本文档根据当前项目已实现的前端页面、后端接口和 SQL Server 表结构整理，用于测试业务功能是否符合预期。

参考文件：

- 前端页面：`frontend/app/page.tsx`
- 后端接口：`backend/app.py`
- 后端资源表映射：`backend/resources.py`
- 数据库结构：`01_schema_sqlserver.sql`
- 模拟数据：`02_seed_mock_data.sql`

## 1. 权限与通用测试前提

| 角色 | 示例账号 | 可见界面 | 主要权限 |
| --- | --- | --- | --- |

### 1.2 通用页面行为

多数业务页面共用同一套表格交互：

| 操作 | 前端行为 | 后端接口 | 通用预期 |
| --- | --- | --- | --- |
| 查询 | 搜索框输入关键字后点击“查询” | `GET /api/{resource}?search=xxx` | 表格只显示匹配记录；总条数同步变化；不修改数据库 |
| 新增 | 点击“新增”，填写表单并保存 | `POST /api/{resource}` | 对应主表增加 1 条记录；相关总览统计可能变化 |
| 编辑 | 点击“编辑”，修改字段并保存 | `PUT /api/{resource}/{id}` | 对应主表指定记录字段变化；相关页面展示同步变化 |
| 删除 | 点击“删除”并确认 | `DELETE /api/{resource}/{id}` | 对应主表记录删除；若存在外键依赖，数据库会拒绝删除 |
| 刷新 | 点击“刷新” | 当前页面 `GET` 接口 | 页面重新加载最新数据 |

| 操作 | 影响表/记录 | 预期受影响界面 |

| 删除车辆 | `t_vehicle` 删除指定车辆 | 若车辆被预约、违规、处罚、黑名单、记分周期等引用，预期删除失败；无引用时车辆档案消失，总览车辆数减少 |
| 查询车牌/类型/状态/备注 | 不修改数据 | 表格返回匹配车辆 |

| 删除预约 | `t_appointment` 删除记录 | 预约入校记录消失；若被 `t_violation.appointment_id` 引用，预期删除失败 |
| 查询车牌/预约方/用途/状态 | 不修改数据 | 表格返回匹配预约 |

### 2.5 违规记录

| 项目 | 内容 |
| --- | --- |
| 主表 | `t_violation` |
| 关联展示 | `t_vehicle.plate_number`、`t_violation_rule.violation_type/violation_level` |
| 页面字段 | 违规 ID、车牌号、违规类型、违规等级、时间、地点、速度、来源、状态、备注 |
| 查询范围 | 管理员查全部；车主仅查本人车辆违规 |

新增“已确认”违规的核心联动：

| 条件 | 自动影响 |
| --- | --- |
| 任意已确认违规且规则扣分大于 0 | 更新/创建 `t_scoring_period`；生成 `t_penalty` 的“扣分”记录 |
| 一次危险违规，扣分大于等于 12 | 生成“暂停入校 60 天”和“全校通报”处罚；更新 `t_vehicle` 为“暂停” |
| 恶性行为规则 | 生成“永久禁止”处罚；更新 `t_vehicle` 为“永久禁止”；新增或刷新 `t_blacklist` 永久黑名单 |
| A 类车累计扣分达到 12/24/36 | 分别生成暂停 15/30/365 天处罚；达到 24 分时还生成“谈话提醒” |
| B 类车单车累计扣分达到 6 | 生成“预约黑名单”处罚；新增或刷新永久黑名单 |
| C 类车累计扣分达到 12 | 生成 365 天暂停入校处罚；更新车辆为暂停 |
| B 类预约车辆违规 | 更新 `t_b_appointer_violation_summary`；单位累计 24 分生成“通报单位”；个人累计 24 分暂停因私预约 15 天；房屋累计 12 分取消房屋预约 30 天 |

操作测试：

| 操作 | 影响表/记录 | 预期受影响界面 |
| --- | --- | --- |
| 新增已确认普通违规 | `t_violation`、`t_scoring_period.deducted_points_total`、`t_penalty` | 违规记录新增；记分周期累计扣分增加；处罚处理新增扣分记录；总览最近违规和违规数量变化 |
| 新增已确认危险违规 | 上述表，另影响 `t_vehicle`，可能影响 `t_blacklist` | 车辆档案状态变“暂停”；处罚处理出现暂停/通报记录；门禁核验禁止入校；总览执行中处罚增加 |
| 新增恶性行为违规 | `t_violation`、`t_penalty`、`t_vehicle`、`t_blacklist` | 车辆档案变“永久禁止”；黑名单新增/刷新；门禁核验禁止入校；处罚处理出现永久禁止记录 |
| 新增 B 类预约违规 | `t_violation`、`t_scoring_period`、`t_penalty`、`t_b_appointer_violation_summary`，可能影响 `t_registrant` 或 `t_house` | 违规记录、处罚处理、记分周期变化；若触发预约方阈值，后续预约审批被拦截 |
| 新增状态为“申诉中/已撤销”的违规 | 仅插入 `t_violation` | 违规记录出现；不应自动扣分或生成处罚 |
| 编辑违规 | `t_violation` 指定记录 | 违规记录字段变化；当前实现编辑不会重新执行自动处罚逻辑 |
| 删除违规 | `t_violation` 删除记录 | 若被申诉、处罚引用，预期删除失败；当前实现删除不会自动回滚已生成扣分/处罚 |
| 查询规则/车牌/类型/地点/来源/状态/备注 | 不修改数据 | 表格返回匹配违规 |

| --- | --- |
| 主表 | `t_penalty` |
| 关联展示 | 来源车辆 `t_vehicle`、目标单位 `t_department`、目标房屋 `t_house` |
| 页面字段 | 处罚 ID、来源车牌、触发类型、处罚描述、开始、结束、状态 |
| 查询范围 | 管理员查全部；车主查本人车辆作为来源/目标或本人作为目标人的处罚 |

操作测试：

| 操作 | 影响表/记录 | 预期受影响界面 |
| --- | --- | --- |
| 新增处罚 | `t_penalty` 新增记录 | 处罚处理新增记录；业务总览执行中处罚可能增加 |
| 编辑处罚状态 | `t_penalty.status` | 处罚处理状态变化；业务总览执行中处罚数量按状态变化 |
| 编辑处罚目标 | `t_penalty.target_vehicle_id/target_dept_id/target_person_id/target_house_id` | 处罚处理目标展示变化；车主可见范围可能变化 |

注意：直接在处罚处理页面新增“暂停入校”“预约黑名单”等处罚，只写入 `t_penalty`，不会自动同步修改 `t_vehicle` 或 `t_blacklist`。自动同步主要发生在新增已确认违规时。

## 4. 重点业务场景建议

### 4.1 A 类车危险违规场景

1. 在违规记录新增一条 A 类车、规则为危险违规、状态“已确认”的记录。
2. 预期 `t_violation` 新增。
3. 预期 `t_scoring_period.deducted_points_total` 增加，`has_danger_violation` 可能变为 1。
4. 预期 `t_penalty` 出现扣分、暂停入校、全校通报等记录。
5. 预期车辆档案中该车状态变为“暂停”，截止日期约为当前日期后 60 天。
6. 预期门禁核验该车为禁止入校。

### 4.2 B 类预约车累计扣分场景

1. 对 B 类车新增已确认违规，使单车累计扣分跨过 6 分。
2. 预期 `t_penalty` 出现“预约黑名单”。
3. 预期 `t_blacklist` 新增或刷新永久黑名单。
4. 预期门禁核验该车为禁止入校。
5. 若违规关联预约，预期 `t_b_appointer_violation_summary` 累计分增加。

### 4.3 房屋预约方处罚场景

1. 对房屋预约的 B 类车新增已确认违规，使房屋年度累计分跨过 12 分。
2. 预期 `t_b_appointer_violation_summary` 对应房屋累计分增加。
3. 预期 `t_penalty` 出现“取消房屋预约”。
4. 预期 `t_house.appointment_status` 变为“暂停”，暂停截止日期约为当前日期后 30 天。
5. 后续该房屋预约审批通过时，预期被拦截。

### 4.4 加分审批通过场景

1. 准备 A 类车活跃记分周期，累计扣分为 12 或 24，年度加分次数小于 2。
2. 车主新增加分申请。
3. 管理员点击“通过”。
4. 预期 `t_points_addition_log.status` 变为“已通过”。
5. 预期 `t_scoring_period.added_points_total` 增加，`add_count` 加 1，剩余分增加。
6. 若该车暂停期已过，预期车辆状态恢复“正常”。

### 4.5 门禁核验场景

1. 查询不存在车牌，预期禁止，原因“未找到车辆档案”。
2. 查询暂停车辆，预期禁止，原因“车辆处于暂停入校状态”。
3. 查询有效黑名单车辆，预期禁止，原因“车辆在有效黑名单中”。
4. 查询 B 类车且当前无有效已通过预约，预期禁止。
5. 查询 B 类车且当前有有效已通过预约，且无暂停/黑名单，预期允许。
6. 查询过期暂停或过期黑名单车辆，预期系统先自动清理，再给出最终核验结果。

## 5. 当前实现中需要特别关注的测试点

| 测试点 | 原因 |
| --- | --- |
| 编辑违规不会重新计算处罚 | 自动处罚逻辑只在新增违规后执行 |
| 删除违规不会回滚扣分和处罚 | 当前删除为单表删除，且受外键约束 |
| 申诉通过不会自动撤销违规/处罚 | 当前申诉处理只更新 `t_appeal` |
| 手动新增处罚不会自动修改车辆或黑名单 | 车辆暂停、黑名单主要由违规自动处理触发 |
| 通知日志不会随违规/处罚自动生成 | 当前通知日志是独立维护 |
| 车主只能新增预约、申诉、加分申请 | 其他资源只读，且按本人 `registrant_id` 过滤 |
| 删除操作可能因外键失败 | 测试时应验证页面是否给出失败提示 |
